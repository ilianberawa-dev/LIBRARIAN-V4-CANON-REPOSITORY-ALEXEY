**Формат блока:**
```
## YYYY-MM-DD HH:MM — <title>
Предложение: <что именно рассылать и кому>
Обоснование: <почему это касается всех>
Статус: pending / approved / rejected
```

## 2026-04-21 18:55 — Продакшн = ТОЛЬКО API, не подписка

**Предложение (от librarian-v2 через PDF 168):** в продакшне (любые client-facing роли) использовать **только API-доступ к LLM**, не подписочный Claude/GPT.

**Обоснование:** подписочные лимиты имеют дневной cap. Когда кончились — агент перестаёт отвечать клиентам посреди работы (например, на обеде у Ильи). Это downtime с прямой финансовой потерей.

**Касается:**
- `secretary-v1` когда начнёт отвечать клиентам брокера (WhatsApp/Email autoreply).
- `linkedin-writer` когда пойдут авто-публикации.
- Любая новая роль обслуживающая внешних пользователей.

**Не касается:** school / librarian / parser — они dev-инструменты Ильи, подписка подходит.

**Канал:** записано в `canon_training.yaml` как под-принцип `role_invariants.production_uses_api_not_subscription` (добавить).

**Статус:** pending_approval (Илья подтверждает → школа пишет принцип в canon + дисratch всем ролям через [CANON UPDATE]).

## 2026-04-21 15:45 — Split-addressing в inbox (TO_SUCCESSOR + TO_SCHOOL)

**Предложение:** каждый inbox-ответ разделён на две секции:
- `## TO_SUCCESSOR` — operational TODO для следующей версии себя (конкретные задачи).
- `## TO_SCHOOL` — архитектурные инсайты для синтеза в canon_training / skills компании.

**Обоснование (от Ильи 2026-04-21):** у сообщений разная аудитория. Successor нужны команды. Школа интегрирует наблюдения в общий канон. Смешивая — теряем качество обоих.

**Рассылка:** всем ролям (parser, librarian, secretary, будущие).
**Канал:** уже записано в `canon_training.yaml` → `role_invariants.split_addressing_in_inbox`. Осталось выслать напоминание через outbox каждой роли.

**Статус:** APPROVED Ильёй 2026-04-21, зашито в canon, requires re-read при следующем turn каждой роли.

## 2026-04-21 15:45 — Real-time push через librarian-бот
