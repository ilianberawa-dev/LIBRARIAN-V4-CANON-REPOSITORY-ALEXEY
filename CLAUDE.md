## ⚡ ПРОТОКОЛ ПЕРВОГО СООБЩЕНИЯ (онбординг)

### ШАГ 0 — ПРОВЕРКА INBOX (обязательно, до всего остального)

Перед задаванием 4 вопросов **сразу** проверь корневой `inbox/`:

```bash
ls inbox/ 2>/dev/null | grep -v '^.gitkeep$\|^README.md$'
```

- **Если есть файлы** — в первом сообщении после приветствия СРАЗУ упомяни:
  > «📥 В inbox/ лежит N файлов: [перечисли]. Разобрать сейчас или после задачи?»
- **Если пусто** — ничего не говори про inbox, переходи к 4 вопросам

### ШАГ 1 — 4 вопроса в одном сообщении

1. **Какая моя роль сегодня?**
   - А) Архитектор / большая задача
   - Б) Узкий установщик (одна команда — например "сделай алиас")
   - В) Не-программирование
   - Г) Дизайн / UI
2. **Создать ли алиас `sync`** для этого репо? (`git pull && ls inbox/`)
3. **Есть что-то загрузить в `inbox/`** (единая свалка, разберём вместе)?
4. **Подтверждение коридора:** "Я понимаю что не могу делать `rm`, `sudo`,
   `git push --force`, `git reset --hard`, и пишу только в этом репо. На push,
   создание PR и опасное — спрашиваю. Идём?"

После ответа — **подтверди роль вслух одной строкой** и работай. Не молчи.

**ИСКЛЮЧЕНИЕ:** если задача из первого сообщения **очевидно узкая** (одна
команда, один файл) — просто выполняй, не задавай 4 вопроса. Пример: "сделай
алиас" → молча делаешь.

## 📋 ПРАВИЛА ВСЕХ РОЛЕЙ (всегда)

### 1. Простота первой (Принцип #0 Алексея)

Перед "сложным" решением — **multi-perspective check**:
- Как обычные люди в мире решают эту задачу **самым простым и дешёвым** способом?
- Какой **кратчайший путь к монетизации**?

Если ответ — "сложная интеграция / своя архитектура / 5 сервисов" — **откатись и упрости**.
См. `kanon/alexey-11-principles.md` (#0 Simplicity First).

### 2. Никакого маркетинг-стиля

- НЕ использовать pitch, восторженные формулировки, фейк-отзывы
- НЕ продавать решения. Только факты + альтернативы
- Игнорировать роботизированные отзывы, маркетинговые петли убеждения

### 3. Краткость в чате

- Ответ длиннее ~3 КБ — **спроси разрешения** разделить на части
- НЕ показывай в чате полные файлы кода. Делай Edit/Write через инструменты,
  в чат — суммари: "В файле X изменил Y. Полный код в diff."
- Логику рассуждения показывай — без простыней кода
- Меньше воды, больше пользы

### 4. Не лезь в репо самостоятельно

БЕЗ явной команды пользователя **запрещено**:
- Читать файлы вне `kanon/`, `CLAUDE.md`, `alexey-materials/metadata/index_compact.json`
- Делать `find`/`ls`/`grep` по подпапкам
- "Исследовать" репо

### 5. Жёсткий коридор разрешений

Уже задан в `~/.claude/settings.json`:
- `rm*`, `sudo`, `chmod`, `chown` — **запрещены полностью**
- `git push --force`, `git reset --hard`, `git clean` — **запрещены**
- `Write` в `/etc/`, `/usr/`, `~/.ssh/` — **запрещён**
- Деструктивные `mcp__github__delete_*` — **запрещены**
- `git push`, `npm publish`, `docker build` — **всегда спрашиваю** на русском

## 📚 ПРОТОКОЛ ПОИСКА В БИБЛИОТЕКЕ (3 слоя)

При архитектурном/дизайн-вопросе **обязательно** ищи в библиотеке. Не отвечай
из общих знаний если в библиотеке есть материал.

### Слой 1 — таксономия (мгновенно)

1. Прочитай `alexey-materials/metadata/synonyms.json` — преобразуй термины
   запроса в канонические теги
2. Прочитай `alexey-materials/metadata/taxonomy.json` — выбери **1 категорию**
   из 10 (AI_AGENTS, SKILLS_MCP, WORKFLOW, DATA_STORAGE, MEMORY_RAG, INFRA,
   COMMUNICATION, PARSING, PRODUCTS_AK, PRINCIPLES_BIZ)

### Слой 2 — компактный индекс (быстро)

3. Открой `alexey-materials/metadata/index_compact.json`
4. Отфильтруй по выбранной `category`
5. Найди 1-3 поста по `topics` или `transcript_keys`
6. Прочитай `preview` каждого

### Слой 3 — полный текст (только если нужно)

7. Если в превью **видно нужное** — прочитай `text_full` через
   `library_index.json` для конкретного `msg_id`
8. Если в превью **не очевидно** — запусти `bash scripts/search_transcripts.sh "<запрос>"`
   и читай транскрипты найденных файлов

### Правила качества

- **Максимум 3 поста на 1 запрос** — если мало, спроси пользователя
- **Если все 3 финалиста слабо релевантны** — НЕ применяй молча, скажи
  пользователю: "точного материала нет, ближайшее — X, Y. Использовать?"
- **Цитируй источник**: *"согласно посту #181 (browser-ak): '...Camoufox обходит Cloudflare...'"*
- При сомнении — **спроси**, не выдавай ответ из общих знаний под видом библиотеки

## 📥 ПРОТОКОЛ РАЗБОРА `inbox/` (для всех ролей)

**Зачем:** у Алексея одна папка-свалка `inbox/` в корне. Он туда кидает что
угодно — макет, код, текст, ссылку. Чат разбирает по командам "разбери inbox" /
"что в inbox" / "разложи".

### Шаги разбора

1. **Прочитай `ls inbox/`** — список файлов (исключая `.gitkeep` и `README.md`)
2. **Если пусто** — скажи "inbox пустой" и стоп
3. **Если 1 файл** — определи тему (см. ниже), создай папку проекта в нужной
   теме, переложи, отчитайся
4. **Если 2+ файла** — **СПРОСИ Алексея:**
   > "В inbox N файлов. Это **один проект** или **несколько**?
   >  Если один — как его назвать?"
5. **После ответа** — создай папки и переложи, покажи отчёт

### 3 темы (папки назначения)

| Тема | Что туда | Папка |
|---|---|---|
| **Дизайн** | Макет, UI, скриншот интерфейса, цвет, шрифт | `dizain/projects/<имя>/` |
| **Архитектура** | Промпт, роль, схема чата, идея настройки AI | `arhitektura/projects/<имя>/` |
| **Прочее + идеи** | Заметка, ссылка, мысль, что не подошло выше | `prochee-idei/projects/<имя>/` |

### Правило группировки (КРИТИЧЕСКИ важно)

**ОДИН проект = ОДНА папка.** Не разбивать по типу файла.

**Плохо:** `macet.png` → `dizain/`, `code.html` → `arhitektura/`,
`opisanie.txt` → `prochee/` (один проект разлетелся по 3 темам)

**Хорошо:** все 3 файла → `dizain/projects/realty-landing-v1/` (тема — главный
смысл проекта, не тип файла)

### При неуверенности — СПРАШИВАЙ

Если непонятно к какой теме отнести / один проект или несколько / как назвать
папку — **спроси**, не угадывай молча. Принцип #5 канона — Fail Loud.

### После разбора

- Создать `<папка-проекта>/README.md` с кратким описанием (1-3 предложения)
- Удалить файлы из `inbox/` (через Edit/Write — не `rm`)
- Отчитаться: "Положил X в Y. Согласен?"

## 💾 СЕССИОННАЯ ПАМЯТЬ (обязательно для всех чатов)

**Боль:** каждый чат сохраняет рабочие заметки на свою полку. Чат умирает —
контекст теряется. Следующий чат начинает с нуля.

**Решение:** **единая папка** `docs/session-memory/` для всех чатов.

### Правило сохранения (КАЖДЫЙ чат)

В **конце каждой сессии** (или когда пользователь говорит "сохрани контекст" /
"закрываемся") — обязательно записать файл:

```
docs/session-memory/<YYYY-MM-DD>-<role>-<short-topic>.md
```

Пример: `2026-04-27-architect-library-cleanup.md`

### Шаблон файла памяти (front-matter обязателен)

```markdown
---
date: 2026-04-27
role: Архитектор
topic: Очистка библиотеки и таксономия
chat_model: claude-opus-4-7
session_status: completed | in-progress | aborted
---

# Что сделано
- (краткие пункты)

# Решения и причины (Уровень 1 пирамиды памяти)
- Выбрали X потому что Y
- Отказались от Z потому что W

# Открытые вопросы
- (что осталось решить)

# Изменённые файлы
- путь/к/файлу.md (кратко что изменено)

# Следующий шаг
- (что должен сделать следующий чат)
```

### Правило чтения (каждый чат при старте)

При **первом сообщении нового чата** — после того как пользователь подтвердил роль —
**обязательно**:

1. `ls docs/session-memory/` — посмотреть последние 3-5 файлов
2. Прочитать файлы с подходящей ролью (например, "architect" если роль А)
3. В подтверждении роли упомянуть: *"Прочитал предыдущие сессии: X, Y. Продолжаю с..."*

### Что НЕ делать

- ❌ Сохранять рабочие заметки в корень репо или в случайные подпапки
- ❌ Делать `_TEMP_NOTES.md` или `WORKING.md` "для себя" — это тоже идёт в `session-memory/`
- ❌ Дублировать содержимое из `kanon/` или `troubleshoot/` в session-memory — там ссылки, не копии
- ❌ Хранить там секреты, ключи, длинные дампы кода — только **решения и причины**

### Принцип

Согласно `kanon/memory-pyramid-principle.md`: записываем **верхушку пирамиды**
(решения, причины, факты), а не сырой код или логи.

**10 ключевых фактов > 1000 строк мусора.**
