## Принцип #0: Simplicity First (добавлен 2024-04-24)

**Правило:** ПЕРЕД ответом: Как обычные люди решают? Есть ли простое стандартное решение? Не усложняю ли я?

**Почему:** AI склонен к overengineering. 2 часа на permissions когда git push = 5 минут.

**Деньги:** Время = деньги. Каждый час на сложное решение = потерянный час на монетизацию.

**Decision tree:** Copy-paste (<100KB) → GitHub/Cloud → Settings (ПОСЛЕДНЕЕ)

**3 правила простоты:**
- **Бабушка-тест:** если решение нельзя объяснить за 30 секунд → ищи проще
- **Счёт шагов:** 1-2 шага ✅ | 3-5 ⚠️ (можно упростить?) | 6+ ❌ (обязательно упрости)
- **Таймер:** <5 мин ✅ | 5-15 мин ⚠️ | >15 мин ❌ (юзер уйдёт)

**Детали:** См. `simplicity-first-principle.md` в этой же папке

## Принцип #1: Portability (Переносимость)

**Правило:** Только официальные Docker-образы. `docker compose up -d` на свежем VPS = всё работает.

**Почему:** Катастрофоустойчивость + дёшево переехать.

**Деньги:** Новый VPS за минуты = нулевой downtime → нулевая потеря revenue.

**Canon ref:** msg_11, msg_13, msg_15, msg_55, msg_136

## Принцип #2: Minimal Integration Code

**Правило:** Вместо кастомного Python/JS — n8n ноды + Skills + MCP серверы. Логика в конфигах, не в обвязке.

**Почему:** Меньше кода → меньше багов → быстрее меняешь.

**Деньги:** Маржа времени на новую фичу ×5. Время = деньги.

**Canon ref:** msg_71, msg_94, msg_95, msg_97

---

## Принцип #3: Simple Nodes

**Правило:** Одна нода/скрипт = одна задача. Не комбайны.

**Почему:** Композиция легко отлаживается, монолит — нет.

**Canon ref:** msg_66, msg_76, msg_101

---

## Принцип #4: Skills Over Agents

**Правило:** Skill > Agent > Hardcoded prompt. Логика в SKILL.md, не в длинных промптах.

**Почему:** Скиллы версионируются, переиспользуются, тестируются. Агент без скиллов — болтун.

**Деньги:** Один SKILL = N-кратное переиспользование через проекты.

**Canon ref:** msg_172, msg_174, msg_178

---

## Принцип #5: Minimal Clear Commands

**Правило:** Императивные короткие инструкции агентам. Fail-loud на плохих входах. Без пространства для галлюцинаций.

**Почему:** Коридор без фантазий. Агент не импровизирует — не ломает продакшн ночью.

**Canon ref:** msg_178

---

## Принцип #6: Single Secret Vault

**Правило:** Все ключи в ЕДИНОМ .env. Никогда в fallback/хардкоде.

**Почему:** Ротация = одно место. Утечка = одно место.

**Пример нарушения:** Ключ LiteLLM в fallback в коде → security risk.

---

## Принцип #7: Offline First

**Правило:** Свой сервер/локалка. Cloud только где реально нужен.

**Почему:** Никто не кинет. Дешевле на масштабе.

**Canon ref:** msg_39, msg_48, msg_158

---

## Принцип #8: Validate Before Automate

**Правило:** Месяц триал → оценить ценность → решить продление. Не вкладываться в автоматизацию на непроверенных данных.

**Почему:** Не автоматизируй то, что не приносит ценность.

---

## Принцип #9: Human Rhythm API

**Правило:** Симулировать человеческое поведение. Случайные паузы 1-5 мин, перерывы 5-20 мин, длинные 30-90 мин. Не бёрстить.

**Почему:** Банят регулярные паттерны. CF блокирует 17/72 страниц на Rumah123 → недобранные контакты.

**Деньги:** Каждый блок = недополученные контакты = недополученные комиссии $10-50k.

**Pattern:** Медленно в одном портале + быстро между разными + случайные длинные перерывы.

**Приоритет:** CRITICAL для parser, librarian, linkedin.

**Эталон:** tg_download_aeza.mjs у librarian

## Принцип #10: Content Factory Model

**Правило:** Парсинг доноров → фильтр по качеству → пересборка в свой стиль → автопубликация.

**Почему:** Масштаб через скиллы, не через найм.

**Canon ref:** msg_101, msg_102, msg_103, msg_104

## Принцип #11: Architectural Privilege Isolation

**Правило:** Критичные права ограничены архитектурно (у агента физически нет инструмента навредить), НЕ промптом.

**Почему:** Prompt injection обходит prompt-based защиту в 90-100% случаев. Единственная надёжная защита — архитектурная.

**Пример плохо:** `supabase-query(sql: '...')` — generic, агент пишет SQL. Injection → DROP TABLE.

**Пример хорошо:** `get-listing-phone(listing_id: int)` — зашитый SQL, только параметр. Injection физически не может DROP.

**Деньги:** Один injection на клиентском агенте = штраф 152-ФЗ до 6 млн руб + потеря клиентской базы.

**Canon ref:** msg_168 (PDF Мультиагентные AI-системы), arXiv:2404.02151

**Applies to:** secretary, future client-facing roles

## Как применять

1. **Перед проектированием системы** — пройтись по всем 12 принципам
2. **При code review** — проверить compliance
3. **При добавлении нового агента** — особенно #11 (privilege isolation)
4. **При проблемах** — найти какой принцип нарушен
