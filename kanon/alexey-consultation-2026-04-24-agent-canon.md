## 2. Почему НЕ Paperclip и мультиагентные системы [ч.2: 02:19 - 05:22]

**Причины отказа:**

- Управлять ордой агентов сложно без опыта постановки задач — качественного результата не добиться
- Для задачи Ильи это **лишний расход токенов**
- **Концепция:** вместо агентов — скиллы. Промты, которые писались бы агентам, просто переезжают в markdown-скиллы + скрипты для детерминированных действий
- **Пример:** раньше DevOps-задачи решались командой из 8 агентов; сейчас тот же функционал — один скилл. Экономия токенов **3-10×**

**Вывод:** мультиагентная система нерентабельна, сложнее в настройке и поддержке, риски выше.

**Связь с каноном:**
- Принцип #4 (Skills Over Agents) — прямое подтверждение
- Принцип #0 (Simplicity First) — меньше компонентов = проще

## 3. Выбор модели для OpenClaw [ч.2: 07:06 - 09:45]

**Ключевой тезис:**
> На OpenClaw (продакшн) тяжёлые модели не нужны — скилл задаёт жёсткий детерминизм

**Модели в продакшне:**

| Модель | Применение | Относительная цена |
|--------|-----------|---------------------|
| **MiniMax M2.7** | Личный OpenClaw Алексея | ~1/15 от Opus |
| **Qwen 3.5** | Клиентские агенты (логистика, автопарки) | Самая маленькая, "прекрасно пашет" |
| **Opus 4.7 / Sonnet 4.6** | Только разработка в Claude Code | Дорого, но нужно для качества скилла |

**Flow разработки:**
1. Пишешь скилл в Claude Code (Opus/Sonnet — нужна сильная модель)
2. Тестируешь руками
3. Закидываешь в OpenClaw
4. OpenClaw запускает скилл на дешёвой модели (MiniMax/Qwen)
5. Ставишь на крон

**Почему это работает:**
Скилл содержит детерминированные инструкции с фиксированными путями. Модели не нужно "думать" — она выполняет чёткий алгоритм. Дешёвая модель справляется не хуже дорогой.

**Связь с каноном:**
- Принцип #5 (Minimal Clear Commands) — "коридор без фантазий"
- Принцип #11 (Architectural Privilege Isolation) — детерминизм через архитектуру, не через промпт

## 5. Архитектура (подтверждение из ч.2) [ч.2: 00:52 - 01:36]

**Стек:**

```
┌────────────────────────────────────────────────┐
│                   CLAUDE CODE                  │
│           (разработка скиллов, Opus/Sonnet)    │
└──────────────────────┬─────────────────────────┘
                       │
                       ▼
┌────────────────────────────────────────────────┐
│                   OPENCLAW                     │
│      (продакшн-runner, MiniMax/Qwen)           │
│      Запускает скиллы по крону или триггеру    │
└──────────┬──────────────────────┬──────────────┘
           │                      │
           ▼                      ▼
┌───────────────────┐   ┌─────────────────────────┐
│     LIGHTRAG      │   │       SUPABASE          │
│   Шина знаний     │   │  Структурированные      │
│  (не числа):      │   │  данные:                │
│  - районы         │   │  - парсинг (листинги)   │
│  - типы объектов  │   │  - логи                 │
│  - специфика      │   │  - задачи агентам       │
│  - тексты статей  │   │  - пользователи         │
└───────────────────┘   └──────────┬──────────────┘
                                   │
                                   ▼
                       ┌─────────────────────────┐
                       │      ФРОНТ + АВТ-ИЯ     │
                       │   Дашборд + OAuth       │
                       │   + Платёжка = SaaS     │
                       └─────────────────────────┘
```

**Распределение:**

| Компонент | Данные | Почему |
|-----------|--------|--------|
| **LightRAG** | Не числа: районы, типы, специфика, описания | Векторный поиск по знаниям |
| **Supabase** | Числа + структурные: парсинг, листинги, задачи, логи | SQL для фронта и аналитики |
| **OpenClaw** | Оркестрация и выполнение | Скилл-раннер на дешёвой модели |
| **Claude Code** | Разработка | Только для написания и тестирования скиллов |

К Supabase потом цепляется фронт (дашборд), авторизация уже есть → прикручиваешь платёжку → **SaaS готов**.

## Применение к Personal AI Assistant (MVP разгрузки чатов)

Этот канон **переопределяет** любые предыдущие архитектурные решения по AI Assistant:

### Что остаётся как было
- SQLite для MVP (не Supabase) — изолированное хранилище личных чатов, не SaaS
- Telegram MTProto listener (gramjs)
- Отдельный bot для inline-кнопок
- Event-driven microkernel + skills
- Heartbeat паттерн

### Что корректируется

**Модели (пересмотр cascade):**

| Задача | Было (моё) | Должно быть (канон) | Комментарий |
|--------|-----------|---------------------|-------------|
| Triage входящих | Haiku 4.5 | **Qwen 3.5** или **MiniMax M2.7** | Детерминированный скилл — дешёвая модель |
| Intent parsing голосовых | Haiku 4.5 | **Qwen 3.5** или **MiniMax M2.7** | То же |
| Draft generation | Sonnet 4.6 | **Sonnet 4.6** OR **MiniMax M2.7** c жёстким skill | Если стиль в skill.md зафиксирован — дешёвая пройдёт |
| Проактивный brief | Sonnet 4.6 | **Sonnet 4.6** | Нестандартный reasoning — оставить |
| Weekly summary | Sonnet 4.6 | **Sonnet 4.6** | То же |

**OpenClaw:**
- НЕ в MVP self-use (чаты) — там нужен простой Node.js listener
- В Phase 2 парсеров — ДА, согласно канону
- В Phase 3 content factory — ДА
- В Phase 4 SaaS для клиентов — ДА (как основной runner)

**Структура скиллов (как у Алексея):**
- Скиллы в `~/.claude/skills/<name>/SKILL.md` — разрабатываются в Claude Code
- Тестируются руками локально
- Деплоятся через OpenClaw на production
- OpenClaw + крон = production runner
