## TL;DR

- **6 этапов, 10-12 дней до production MVP**
- **Один мозг (Sonnet 4.6) + Grok STT + SQLite + FTS5** — никаких лишних сервисов
- **~$20/мес** на API, hard cap через native budget guard
- **Только Telegram** (PWA в Phase 2)
- **Incremental ingest** (no full 3-year history в MVP)
- **Manual learning** первые 2 недели (автомат только в Phase 2)

## Цель

**Боль владельца (Алексей/Илья):**
- Перегруз каналами (3 WhatsApp + 3 Telegram + 4 email)
- Не отвечает важным, забывает ДР, пропускает новости
- Нужна проактивная система, не падает (heartbeat), умнеет со временем

**Метрика успеха:**
1. **Ноды не падают** — uptime ≥99% за 7 дней без ручного вмешательства
2. **Машина умнеет** — процент правок пользователя в драфтах снижается неделя к неделе

**Фаза методологии:** Self-Use (фаза 2 из `v1.0-mvp-to-sales.md`)

## Этап 0 (one-time): Dev Workflow Setup (1 день, делает владелец)

**Что делает владелец на своей машине:**

Настраивает в Claude Code Max три глобальных sub-agents (по канону Алексея — `kanon/alexey-realty-parser-canon-2026-04-24.md`). Это НЕ часть production стека, это **dev workflow** на ноуте — экономит ~80% токенов при разработке скиллов.

```
~/.claude/agents/
├── scout-haiku.md      — Haiku 4.5: поиск файлов, FTS5 запросы, справки (дёшево)
├── worker-sonnet.md    — Sonnet 4.6: дефолтный исполнитель, пишет скиллы
└── strategist-opus.md  — Opus 4.7: архитектура, планы, трейдоффы (редко)
```

**CLAUDE.md** в корне проекта роутит задачи нужному sub-agent'у. 80% задач уйдёт на Haiku (в 15× дешевле Opus).

**Канон-чек:** соответствует второй консультации Алексея (realty parser canon), глобальные субагенты обязательны для всех его клиентов.

**Критерий приёмки:** в Claude Code чате команда "найди X в проекте" автоматически направляется на scout-haiku, "напиши скилл Y" — на worker-sonnet.

## 6 этапов MVP (после Этапа 0)

### Этап 1. Listener live (1-2 дня)

**Что работает:**
- MTProto listener на твой аккаунт
- Каждое входящее → SQLite + форвард в канал AI Assistant
- Формат: `📩 17:42 от Илья Иванов: "Привет, ты в офисе?"`

**Техтребования:**
- `/opt/personal-assistant/listener.mjs` (gramjs, event `NewMessage`)
- `/opt/personal-assistant/assistant.db` со схемой (см. ниже)
- `/opt/personal-assistant/.env` с секретами
- systemd unit `personal-assistant-listener.service`

**SQLite schema v1:**

```sql
CREATE TABLE contacts (
  tg_id INTEGER PRIMARY KEY,
  name TEXT,
  username TEXT,
  first_seen TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  last_msg_at TIMESTAMP,
  msg_count_30d INTEGER DEFAULT 0,
  priority TEXT,  -- hot/regular/new/noise/vip (computed)
  is_vip BOOLEAN DEFAULT 0,
  notes TEXT,
  tone TEXT
);

CREATE TABLE messages (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  tg_msg_id INTEGER,
  chat_id INTEGER,
  from_id INTEGER REFERENCES contacts(tg_id),
  text TEXT,
  received_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  category TEXT,  -- question/fyi/promo/spam/social
  urgent BOOLEAN DEFAULT 0,
  handled BOOLEAN DEFAULT 0
);

CREATE VIRTUAL TABLE messages_fts USING fts5(
  text, from_name, chat_name,
  content=messages,
  tokenize='unicode61 remove_diacritics 2'
);

CREATE TABLE drafts (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  msg_id INTEGER REFERENCES messages(id),
  draft_text TEXT,
  generated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  verdict TEXT,  -- sent_as_is/edited/rejected/pending
  final_text TEXT,
  feedback_note TEXT
);

CREATE TABLE rules (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  scope TEXT,  -- contact/category/chat
  scope_id TEXT,
  action TEXT,  -- never_draft/always_urgent/summarize_only/mute
  note TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE voice_samples (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  to_contact_id INTEGER REFERENCES contacts(tg_id),
  text TEXT,
  sent_at TIMESTAMP
);

CREATE TABLE budget_log (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  date DATE,
  input_tokens INTEGER,
  output_tokens INTEGER,
  cost_usd REAL,
  operation TEXT
);
```

**Canon check:**
- #1 Portability ✅ (systemd + SQLite file + .env)
- #2 Minimal Integration ✅ (gramjs готовый)
- #6 Single Vault ✅
- #11 Privilege Isolation ✅ (отдельная папка, своя БД)

**Критерий приёмки:** отправь себе сообщение → через 3 сек появилось в канале AI Assistant + запись в SQLite

### Этап 2. Triage (1 день)

**Что работает:**
- Каждое входящее классифицируется: `hot / regular / new / noise + urgent?`
- В канале форвард идёт уже с тэгами:
  - `📩 17:42 🔥 HOT 🚨 URGENT — от Илья: "Срочно позвони"`
  - `📩 17:45 🗑 NOISE — Канал "Новости..."`

**Техтребования:**
- `/opt/personal-assistant/triage.mjs`
- `/opt/personal-assistant/skills/triage.md` — SKILL с детерминированным промптом
- Anthropic SDK (sonnet-4-6) через одну функцию-обёртку
- Prompt caching на system prompt (skill)

**Decision flow:**

```
incoming →
  is_bot_or_channel? → noise (no LLM call)
  contact.priority == 'vip' && text contains 'срочно|?|help' → urgent=1, hot
  contact.msg_count_30d > 10 → hot
  contact.msg_count_30d > 3 → regular
  contact.msg_count_30d < 3 → new
  → Sonnet call with skill: classify category (question/fyi/promo/social/spam)
```

**Canon check:**
- #0 Simplicity First ✅ (rules first, LLM last)
- #3 Simple Nodes ✅
- #4 Skills Over Agents ✅
- #5 Minimal Clear Commands ✅ (детерминированный skill)

**Критерий приёмки:** на 100 реальных сообщений ≥85% классификация визуально правильная

### Этап 3. Drafts (2 дня)

**Что работает:**
- Для каждого важного входящего (hot/regular + question) — draft в канал
- Формат:
  ```
  📩 17:42 🔥 HOT — Илья: "Встреча в 12?"
  💡 Draft: "Да, в офисе как договаривались"
  ```

**Техтребования:**
- `/opt/personal-assistant/draft_gen.mjs`
- `/opt/personal-assistant/skills/draft.md` — SKILL с voice_profile
- Контекст: последние 5 сообщений с этим контактом + contact.tone + contact.notes
- Budget guard: если `MONTHLY_SPEND * 1.1 > CAP` — skip draft, flag для ручного

**Budget guard (20 строк):**

```javascript
async function spend(tokens, cost) {
  const thisMonth = await db.get("SELECT SUM(cost_usd) as c FROM budget_log WHERE date >= date('now', 'start of month')");
  if (thisMonth.c > CAP * 0.9 && !alertedThisWeek) {
    notifyTelegram(`⚠️ Budget 90%: ${thisMonth.c}/${CAP}`);
    alertedThisWeek = true;
  }
  if (thisMonth.c >= CAP) {
    notifyTelegram(`🚨 BUDGET CAP HIT — drafts disabled`);
    throw new BudgetExceeded();
  }
  await db.run("INSERT INTO budget_log ...", [tokens, cost]);
}
```

**Canon check:**
- #4 Skills Over Agents ✅ (логика в draft.md, не в коде)
- #0 Simplicity First ✅ (один model, native SDK)

**Критерий приёмки:** на 20 реальных входящих, минимум 60% драфтов можно отправить без правок

### Этап 4. Inline buttons bot (1 день)

**Что работает:**
- Второй бот (@your_assistant_bot) шлёт briefs в канал с кнопками
- Клик [✅ Отправить] → через MTProto уходит собеседнику
- Клик [✏️ Правка] → бот просит ввести текст
- Клик [🎙 Голосом] → TTS → voice (Phase 2, пока только плейсхолдер)
- Клик [🚫 Игнор] → verdict=rejected

**Техтребования:**
- `/opt/personal-assistant/bot.mjs` (node-telegram-bot-api)
- callback_query handler → action router → MTProto send
- Rate limit: не более 20 отправок/час через кнопку

**Canon check:**
- #11 Privilege Isolation ✅ (бот НЕ имеет MTProto доступа напрямую, только через router)
- #3 Simple Nodes ✅

**Критерий приёмки:** 5 кнопок работают end-to-end, отправка реально доходит до адресата

### Этап 5. Voice — 3 команды (1-2 дня)

**Что работает:**
Ты шлёшь voice message в @your_assistant_bot → STT → intent → action

**3 intent'а (максимум в MVP):**

1. **"Ответь [кому]: [текст]"** → создаёт draft в чат с указанным контактом
2. **"Правило [контакт/группа] [never_draft/vip/mute]"** → INSERT в rules
3. **"Поиск [запрос]"** → FTS5 + top 5 results в канал

**Техтребования:**
- `/opt/personal-assistant/voice.mjs`
- Переиспользует `/opt/tg-export/transcribe.sh` логику, но адаптирует — добавляет `language=ru` параметр
- Intent parser: Sonnet один вызов с skill
- Fail-loud: если intent не распознан → "Не уловил. Примеры: ..."

**Canon check:**
- #2 Minimal Integration ✅ (Grok STT уже есть)
- #5 Minimal Clear Commands ✅ (3 intent'а, не 15)

**Критерий приёмки:** каждая из 3 команд работает 90%+ на твоём голосе

### Этап 6. Heartbeat + monitoring (1 день)

**Что работает:**
- Cron: day 5 min / night 30 min (по ранее принятому плану)
- Проверяет: listener жив, bot жив, SQLite доступна, бюджет не превышен
- Локальный alert в канал при проблеме
- Healthchecks.io ping снаружи — если сервер упал, узнаёшь из hc-ping notification

**Техтребования:**
- `/opt/personal-assistant/heartbeat.sh` — адаптирован из `navyki/heartbeat-telegram.sh` с исправлениями:
  - Schedule day/night
  - Debounce (1 alert per тип/час)
  - Log-freshness check (не только pgrep)
  - Timer обновляется ТОЛЬКО после успешного триггера
  - TZ из .env
  - Убраны `claude-skill` placeholder'ы
- `HCPING_URL` в .env — бесплатный account на healthchecks.io

**Дополнительно — SQLite daily backup (профилактика поломок):**

```bash
# /opt/personal-assistant/backup.sh — отдельный cron daily 04:00
sqlite3 /opt/personal-assistant/assistant.db ".backup /opt/personal-assistant/backups/$(date +%F).db"
find /opt/personal-assistant/backups/ -mtime +7 -delete
```

Cron: `0 4 * * * /opt/personal-assistant/backup.sh`

**Зачем:** SQLite single-file = corruption риск. Backup'ы 7 дней = ~100 MB на диске. Опционально rsync в облако.

**Canon check:**
- #5 Fail Loud ✅
- #9 Human Rhythm ✅ (jitter в cron)
- Профилактика поломок ✅ (SQLite backup)

**Критерий приёмки:** 72 часа uptime, все auto-restart'ы отработали, 0 false positives, backup-файлы появляются ежедневно
