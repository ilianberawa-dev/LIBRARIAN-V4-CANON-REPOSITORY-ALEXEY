# Orchestration Lessons Learned — Stage 1 file provenance failure

**Дата:** 2026-04-25
**Статус:** RETROSPECTIVE — для предотвращения регрессии в Stage 2-6
**Происшествие:** работяга #1 запущен с ссылкой на несуществующие локальные файлы

---

## Что произошло

Архитектор передал прорабу metadata о 6 файлах Stage 1, упомянутых как "написанные предыдущим установщиком на dev машине владельца". Прораб через recovery JSON получил только metadata + статусы, но не content. Работяга в Linux окружении не имел доступа к Windows-пути dev машины.

**Финальное последствие:** работяга съел токены пытаясь разобраться, прораб самостоятельно сгенерировал файлы по канонической спеке.

---

## Root cause (5 уровней)

1. Файлы предыдущей сессии **никогда не пушились в git репо**
2. Архитектор передал прорабу **metadata вместо content**
3. Recovery JSON-агент **не возвращал поле content** по умолчанию
4. Прораб **поверил metadata** без верификации существования файлов в работающем окружении
5. Промпт работяги ссылался на **dev-машину владельца** (Windows path C:\), не на git repo

**Главный root cause:** отсутствие инварианта "git = single source of truth для всех файлов проекта".

---

## Установленные инварианты (закрепить в FOREMAN-PROMPT.md)

### ИНВАРИАНТ #1: Git repo = single source of truth

```
Ни один файл не существует для Stage пока не закоммичен в репозиторий
ilianberawa-dev/LIBRARIAN-V4-CANON-REPOSITORY-ALEXEY на ветку проекта.

Recovery agents, JSON-блоки в чате, dev-машины владельца,
ссылки на C:\... — НЕ valid sources.

Прораб БЛОКИРУЕТ GO работяге пока:
1. Все файлы Stage находятся в metody/personal-ai-assistant/implementation/stage-N/
2. Файлы видны через mcp__github__get_file_contents
3. Работяга подтвердил что может прочитать их через git clone или GitHub MCP

ЕСЛИ файлы поступили в виде recovery JSON / metadata-only /
ссылок на dev-машину:
- Прораб обязан запросить content явно
- ИЛИ сгенерировать сам по канонической спеке
- ОБЯЗАТЕЛЬНО закоммитить в репо ДО передачи работяге
```

### ИНВАРИАНТ #2: Архитектор не передаёт metadata вместо content

```
Архитектор НЕ может ссылаться на файлы без подтверждённого content в репо.

Если архитектор видит "файлы написаны в прошлой сессии локально":
- Запросить у владельца git push этих файлов в репо
- ИЛИ потребовать у прораба генерации с нуля по канонической спеке
- НЕ передавать metadata вниз по цепи orchestration

Метка проверки: перед approval любого промпта работяге архитектор
проверяет через mcp__github__get_file_contents что файл существует.
```

### ИНВАРИАНТ #3: Recovery agents возвращают content

```
Если используется recovery агент для проверки файлов:
- Запрос ДОЛЖЕН включать поле "content" по каждому файлу
- Если recovery вернул только metadata — это неполный отчёт, требовать повтора
- Прораб не принимает recovery JSON как source of truth без content
```

### ИНВАРИАНТ #4: Промпт работяги ссылается только на репо

```
Промпт работяги Stage N ОБЯЗАТЕЛЬНО:
1. Содержит git clone команду на ветку проекта
2. Указывает путь metody/personal-ai-assistant/implementation/stage-N/
3. НЕ ссылается на C:\..., /home/owner/..., локальные пути dev-машины
4. НЕ содержит JSON-блоков с file content (использует git вместо JSON)

Если файлы переданы в виде JSON-блока — это аварийный workaround,
сразу после прораб ОБЯЗАН закоммитить в репо и обновить промпт.
```

## Стоимость инцидента

- **Токены работяги:** потрачены на попытки разобраться с несуществующими C:\ файлами + парсинг recovery JSON
- **Время прораба:** несколько часов на root-cause analysis + регенерация 6 файлов
- **Время архитектора:** ревью эскалации
- **Календарное время:** ~1 день задержки Stage 1
- **Ущерб качеству:** ноль (acceptance test всё равно отловит регрессии)

---

## Что НЕ делать в Stage 2-6

❌ Ссылаться на локальные файлы dev-машины
❌ Принимать recovery JSON без поля content
❌ Передавать metadata вниз по цепи orchestration без content
❌ Запускать работягу пока файлы не в git репо
❌ Использовать JSON-блоки в чате как source of truth (только аварийный transit)

## Что делать в Stage 2-6

✅ Все файлы Stage коммитятся в `implementation/stage-N/` ДО передачи работяге
✅ Промпт работяги начинается с git clone инструкции
✅ Архитектор перед approval промпта проверяет существование файлов через GitHub MCP
✅ Прораб блокирует GO пока файлы не верифицированы
✅ Recovery agents запрашиваются с обязательным полем content

## Конкретные паттерны для будущих Stage

### Pattern 1: Регенерация файлов прорабом (когда нет источника)

```
ЕСЛИ архитектор: "файлы для этапа есть локально/в прошлой сессии"
И git repo не содержит этих файлов
ТО:
  1. Прораб генерирует файлы по канонической спеке INSTALLATION-PLAN.md
  2. Прораб коммитит в implementation/stage-N/ через GitHub MCP
  3. Прораб обновляет промпт работяги ссылкой на репо (не на чат)
  4. Прораб уведомляет архитектора одной фразой "Files for Stage N committed at <SHA>"
```

### Pattern 2: Git clone в первом действии работяги

```
В промпт работяги обязательно включить первым шагом:

git clone -b claude/setup-library-access-FrRfh \
  https://github.com/ilianberawa-dev/librarian-v4-canon-repository-alexey \
  /tmp/repo

cp /tmp/repo/metody/personal-ai-assistant/implementation/stage-N/* /tmp/stage-N/

Проверка: ls /tmp/stage-N/ должен показать ожидаемое количество файлов.
Если файлов нет — STOP, эскалация прорабу.
```

### Pattern 3: Архитектор pre-flight check через GitHub MCP

```
Перед approval промпта работяги Stage N архитектор обязан:

1. mcp__github__get_file_contents(
     path: "metody/personal-ai-assistant/implementation/stage-N/"
   )
2. Убедиться что все ожидаемые файлы присутствуют
3. Только тогда давать GO прорабу

Если файлов нет — НЕТ APPROVAL, прораб сначала коммитит.
```

## ИНВАРИАНТ #5 (добавлен 2026-04-25 после второго timeout)

```
БОЛЬШИЕ PAYLOAD — ОТДЕЛЬНЫМ СООБЩЕНИЕМ

Любой turn в чате (архитектор, прораб, работяга) который содержит:
- JSON блок >2 KB с file content
- Markdown спецификацию >100 строк
- Multi-tool call с 3+ Write/Edit операциями
- Полный transcript / dump файлов

ОБЯЗАТЕЛЬНО разбивается на:
1. Короткое preamble сообщение (<500 символов): что будет дальше
2. Отдельный turn с самим payload
3. Опциональный finale: что делать с payload

ЗАЧЕМ:
Stream idle timeout API ломает turn если генерация занимает >30 сек.
Длинный preamble + большой tool call в одном turn = высокий риск timeout.
Потеря turn = потеря контекста + работа повторно.

ПРАВИЛО АРХИТЕКТОРА:
Если знаешь что turn будет содержать большой payload — first turn это
preamble "сейчас пришлю X отдельным сообщением", second turn — payload.

ПРАВИЛО ПРОРАБА:
JSON-блоки для работяг — отдельным сообщением, не в составе длинного
объяснения. Расчёт времени по количеству строк × 0.05 сек.

ПРАВИЛО РАБОТЯГИ:
Mapping report = отдельный turn от любых тестов или logs дампов.
```

## ИНВАРИАНТ #6 (добавлен 2026-04-25 после прораба-таймаутов)

```
КРАТКИЕ ОТВЕТЫ ПРОРАБА — ОБЯЗАТЕЛЬНОЕ ПРАВИЛО

Прораб формирует ОТВЕТЫ архитектору и владельцу по принципу:
максимум 5-10 строк на стандартные ситуации, развёрнутый только при ESCALATION.

ПОЧЕМУ:
- Длинные turns >30 сек генерации = риск stream idle timeout
- Многословие маскирует реальный статус
- Работа прораба ≈ оркестрация, не writing

ШАБЛОНЫ КРАТКИХ ОТВЕТОВ ПРОРАБА:

1. После approval архитектора:
   "Принято. Файлы коммичу в implementation/stage-N/, отчитаюсь SHA."

2. После генерации файлов:
   "Stage N files committed at <SHA>. Worker prompt отдельным сообщением."

3. После mapping report от работяги (PASS):
   "Stage N PASS. Acceptance: [3-5 фактов]. Готов к Stage N+1."

4. После mapping report от работяги (FAIL):
   "Stage N FAIL on [criterion]. Logs: [1 cmd output]. Эскалация ниже."

5. На вопрос владельца "что сейчас":
   "Stage N: [одна фраза статуса]. Следующее: [одно действие]."

КОГДА ДЛИННЫЙ ОТВЕТ ОПРАВДАН (исключения):

- ESCALATION к архитектору с root cause analysis (макс 200 слов)
- Pre-flight gate report (5-7 строк checklist)
- Worker prompt JSON (отдельным turn, не считается "ответом")

ЗАПРЕТЫ:
❌ Препоратические таблицы в каждом ответе
❌ Цитаты канона если архитектор уже сослался
❌ Повторение задачи которую только что получил от архитектора
❌ Multi-tool calls в одном turn если можно одним
❌ Ответ "ок" + 200 слов рассуждений

ИСПОЛНЕНИЕ:
Если ответ прораба >15 строк (кроме исключений выше) — это нарушение #6.
Архитектор может вернуть с пометкой "shorten please".
```

## ИНВАРИАНТ #7 (добавлен 2026-04-25 после teleport discovery)

```
TELEPORT WORKFLOW МЕЖДУ УРОВНЯМИ ORCHESTRATION

Команда (PowerShell/bash):
  cd <project-root>
  claude --teleport <session_id>

Применение:
- L0 (архитектор) → L-1 (прораб): передача архитектурного контекста
- L-1 (прораб) → L-2 (работяга): передача mission + готовых файлов
- L-2 → L-1: передача mapping report для acceptance review
- Любой уровень → tools/recovery agent: передача узкой задачи

Преимущества vs новый чат:
- Сохраняется полная история контекста (token waste = 0)
- Не нужен повторный onboarding (canon, файлы, инварианты)
- Быстрее (5 сек vs 5 минут на чтение mandatory_reading)

Требования (Windows):
- Claude Code установлен в user-space (npm i -g без admin) ИЛИ
- PowerShell запущен как Administrator (UAC prompt разовый — OK подтвердить)

Требования (Linux/macOS):
- Любой shell, без admin

Когда НЕ teleport (использовать новый чат):
- Контекст текущей сессии испорчен (>80% токенов, ошибки)
- Нужен чистый старт без legacy решений
- Работяга закончил миссию — закрываем чат, не сохраняем

Альтернатива (если teleport недоступен):
- Новый Claude Code чат
- git clone -b claude/setup-library-access-FrRfh репо
- Прочитать mandatory_reading из FOREMAN-PROMPT.md или INSTALLER-TEMPLATE.md
- Получить mission/spec через JSON или git path

ВКЛЮЧИТЬ В ОБУЧЕНИЕ:
- FOREMAN-PROMPT.md — раздел "Teleport workflow"
- INSTALLER-TEMPLATE.md — раздел "Receiving teleport from foreman"
- Архитектор использует при передаче L-1 (если прораб тормозит, telepport state)
```
