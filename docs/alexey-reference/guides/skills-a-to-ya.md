Зачем этот гайд
Скиллы — самый недооценённый инструмент в AI-автоматизации. Все говорят про
промпты,  MCP-серверы,  мультиагентные  архитектуры  —  а  потом  один
Markdown-файл решает задачу, на которую раньше уходила команда из 8 агентов.
Этот  гайд  —  не  теория  из  документации.  Здесь  конкретные  архитектуры,
реальные кейсы и рабочие паттерны. От простого текстового скилла на 30 строк
до production-системы с 15 скриптами и 10 reference-файлами.
Если ты никогда не создавал скилл — после этого гайда создашь.
Если уже создавал — узнаешь как сделать его стабильнее, экономичнее и
автономнее.
Встретил незнакомый термин? Все определения — в Словаре терминов в
конце гайда.

Часть I — Как мы к этому пришли
1. 8 агентов, 8 точек отказа
Полгода назад перед нами стояла задача: автоматизировать настройку серверов.
Купил VPS, нужно поставить Docker, Caddy, n8n, Supabase, настроить DNS, SSL, 
файрвол, сгенерировать отчёт с логинами.
Единственный инструмент управления моделью в то время — промпт. Хочешь
сложную автоматизацию — пиши несколько агентов, каждый со своим промптом,
и координируй их вручную.
Мы так и сделали. Собрали систему из 8 DevOps-агентов в Kilo Code:
Агент Задача
base-server-setup Базовая настройка сервера, сбор данных
n8n-deployment Деплой n8n + PostgreSQL
supabase-deployment Деплой Supabase
cloudflare-dns-manager DNS-записи в Cloudflare
nginx-proxy-configurator SSL-проксирование для всех сервисов
system-validator Проверка что всё работает
final-report-agent Финальный отчёт для клиента
devops-chain-orchestrator Координирует остальных семерых
8 промптов. 8 конфигураций. 8 точек, где что-то может пойти не так. И это только
для двух сервисов — n8n и Supabase.
Что пошло не так:
Контекст терялся.Оркестратор передавал задачу DNS-агенту, тот делал
записи — но когда результат возвращался обратно, оркестратор уже "забыл"
часть предыдущих шагов. Контекстное окно не резиновое.
Координация жрала токены. Половина токенов уходила не на работу, а на
общение между агентами: "передай DNS-агенту", "получи результат от DNS-
агента", "проверь что DNS-агент закончил".
Хрупкость. Один агент упал → цепочка сломалась. Перезапустить с середины
невозможно — нужно начинать сначала.

Часть II — Что такое скилл и как
он устроен
3. Скилл — это просто файл
Скилл — это текстовый файл с пошаговыми инструкциями. Буквально Markdown-
документ, в котором написано: "когда попросят X — делай Y, потом Z, проверь
результат".
Как рецепт: шаг 1, шаг 2, шаг 3. Только вместо блюда на выходе — настроенный
сервер, составленный отчёт, обработанная заявка.
Минимальный скилл — одна папка с одним файлом:
my-skill/
  SKILL.md    ← весь скилл здесь
SKILL.md состоит из двух частей:
---                              ← YAML-метаданные (имя + описание)
name: my-skill
description: "Что делает. Когда использовать. Ключевые слова."
---                              ← конец метаданных
# Название                       ← дальше Markdown-инструкции
## Когда активировать
## Пошаговые инструкции
## Примеры результата
## Правила
Отличие от промпта:

Промпт загружает все инструкции разом. Скилл загружает их поэтапно — сначала
описание ("нужен ли я?"), потом инструкции ("что делать?"), потом детали по
конкретной подзадаче.
Представь: ты открыл книгу на 300 страниц и пытаешься удержать в голове всё
одновременно. Или ты открыл оглавление, нашёл нужную главу и читаешь только
её. Скилл — это второй вариант.
Скилл  —  это  не  одно  действие. Один  скилл  может  объединять  десятки
операций в один процесс, в том числе обращаясь к разным внешним сервисам и
API:  забрать  данные  из  базы  (Supabase),  найти  контекст  в  базе  знаний
(LightRAG), сгенерировать PDF-отчёт, отправить уведомление в Telegram — и всё
это по одной команде. Каждый шаг прописан, порядок зафиксирован, агент идёт
по инструкции от начала до конца.
Скилл  не  делает  модель  умнее. Модель  и  так  знает,  как  писать  Docker
Compose, составлять контент-план или анализировать договор. Скилл говорит:
"делай  это  в  ТАКОМ  порядке,  с  ТАКИМИ  параметрами,  проверяй  ТАК".
Направляет, а не обучает.
Когда скилл нужен, а когда нет:
Делаешь что-то чаще 3 раз → пиши скилл
Делаешь один раз → просто напиши в чат
Задача требует стабильного, повторяемого результата → скилл
Задача разовая и творческая → обычный промпт
Где скиллы работают:
Единый стандарт (Agent Skills). Один и тот же SKILL.md работает в Claude Code,
Gemini CLI, Codex CLI, Perplexity Computer, OpenClaw и 20+ других платформах.
Написал один раз — запускай где угодно.

4. Три уровня сложности: от рецепта до конвейера
Скиллы масштабируются. Начни с простого — усложняй когда нужно.
Уровень 1: Просто SKILL.md
Один  файл,  20-50  строк.  Подходит  для  80%  задач:  консультации,  генерация
контента, анализ, подготовка документов.
---
name: nutrition-advisor
description: "Составление рациона питания и рекомендации
  по нутрициологии. Используй когда: питание, диета, рацион,
  калории, БЖУ, что поесть, здоровое питание, nutrition."
---
# Консультант по питанию
Отвечай на русском.
## Алгоритм
1. Спроси цель: похудение / набор массы / поддержание / здоровье
2. Спроси ограничения: аллергии, непереносимости, вегетарианство
3. Спроси текущий режим: сколько приёмов пищи, что обычно ешь
4. Рассчитай норму КБЖУ по формуле Миффлина-Сан Жеора
5. Предложи 3 варианта дневного рациона
## Правила
- НЕ ставь диагнозы
- При серьёзных проблемах → направь к врачу-диетологу
30 строк. Ноль скриптов. Работает.
Уровень 2: SKILL.md + references/
Когда  информации  много  —  выноси  детали  в  отдельные  файлы.  SKILL.md
становится  маршрутизатором: определяет тип запроса и отправляет к нужному
reference.
content-plan/
├── SKILL.md                    ← маршрутизатор (50 строк)
└── references/

    ├── tone-checklist.md       ← чеклист тона и стоп-слов

Уровень 3: SKILL.md + references/ + scripts/
Когда  нужна  предсказуемость  и  повторяемость:  установка  софта,  генерация
документов, обращение к внешним API.

├── references/
│   ├── documents-checklist.md  ← список документов по категориям
│   └── troubleshooting.md      ← частые проблемы
└── scripts/
    ├── create-profile.sh       ← создание профиля в системе
    ├── check-fleet-api.sh      ← проверка статуса в API перевозчика
    └── send-welcome.sh         ← отправка welcome-сообщения
Скрипты  делают  то,  что  модель  каждый  раз  делала  бы  чуть  по-разному:
обращение к API, генерация credentials, создание записей в базе. Один и тот же
скрипт, одни и те же параметры — протестировано 50 раз.
Есть ещё assets/ — шаблоны для результата. Отличие от references: references
— инструкции "как делать" (для модели), assets — заготовки "из чего делать"
(шаблоны отчётов, конфиги, которые модель заполняет данными).
Начинай с уровня 1. Большинству задач хватает одного SKILL.md.
Усложняй только когда реально упрёшься в ограничения.

5. Почему контекст — это всё
Весь дизайн скиллов построен на одной идее: контекстное окно модели — это
её оперативная память, и она ограничена.
Вот что происходит когда модель работает:
Контекстное окно (допустим, 200K токенов):
Что занимает место Токены
Системный промпт + конфиг платформы ~5 000
SKILL.md (после активации) ~2 000
Reference-файл (если нужен) ~500
История диалога с пользователем ~10 000
Результаты команд и работы ~50 000
СВОБОДНОЕ МЕСТО для работы ~132 500
Если запихнуть все инструкции сразу — токены потрачены впустую. На длинной
сессии это реально упирается в потолок.
Три принципа, которые мы вывели:
Принцип 1: Загружай только то, что нужно прямо сейчас.
SKILL.md = маршрутизатор (~2000 токенов). References = детали (~500 токенов
каждый). Загружаются по одному. Модель видит только те инструкции, которые
ей нужны на текущем шаге.
Принцип 2: Не объясняй модели то, что она уже знает.
Модель знает, что такое Docker, как написать compose-файл, как составить SWOT-
анализ. Не трать токены на учебник. Пиши только СВОЮ специфику: порядок
действий, конкретные параметры, формат результата, бизнес-правила.
# Плохо (модель и так это знает):
Docker Compose — это инструмент для определения и запуска
многоконтейнерных Docker-приложений...
# Хорошо (конкретика, которую модель не знает):
Порядок установки:
1. setup-server.sh → Docker, UFW
2. install-caddy.sh → reverse proxy, сеть infra

3. cloudflare-dns.sh → A-записи
4. LightRAG → ПОСЛЕДНИМ
Принцип 3: Каждый токен должен оправдывать своё присутствие.
Если  убрать  фрагмент  инструкции  и  результат  не  ухудшится  —  убирай.  Чем
меньше шума в контексте, тем точнее модель выполняет задачу.
Про инструменты: меньше = лучше
Если скилл работает в среде с кучей  MCP-серверов и инструментов — модель
тратит токены на выбор между ними. Несколько правил:
Минимальный набор. Не подключай инструменты, которые скиллу не нужны.
Задача про генерацию контента — не нужны инструменты для работы с базами
данных.
Понятные описания. Модель выбирает инструмент по его описанию. Плохо: 
tool_1. Хорошо: extract_text_from_pdf.
Минимальное перекрытие. Два инструмента с похожими описаниями путают
модель — она не знает какой выбрать.
Компактные результаты. Инструменты должны возвращать
структурированный ответ, а не сырой дамп на 500 строк.
Что происходит когда контекст забивается
Мы это прочувствовали на devops-ak. Полная установка сервера — это 6 фаз, 6-8
сервисов, десятки  bash-команд. К финальной фазе контекст забит: SKILL.md +
несколько  references  +  результаты  30+  команд  +  весь  диалог.  Начинается
деградация:
Модель "забывает" ранние инструкции. Правило "NEVER delete without
confirmation" утонуло в сотнях строк вывода.
Модель пропускает шаги. Из 8 скриптов помнит что запустила 5, хотя
запустила 7.
Формат ответов плывёт. В начале — структурированный вывод, к концу —
поток текста.
Как мы это решили:
TodoWrite для трекинга фаз. Модель создаёт чеклист в начале и отмечает
каждую фазу. Даже когда инструкции "тонут" — чеклист всегда на виду.
Минимизация вывода. Скрипты пишут JSON-лог в файл, а в stdout — только
статус: "OK: n8n installed" или "ERROR: port busy". Не дампят весь compose в
контекст.

6. Где живут скиллы
Глобальные и проектные
Глобальные — доступны всегда, из любого проекта:
~/.claude/skills/my-skill/SKILL.md
Для универсальных скиллов: ответы на отзывы, генерация КП, конспекты встреч
— всё что нужно независимо от того, в какой папке ты сейчас работаешь. Открыл
терминал где угодно — скилл доступен.
Проектные — только в рамках конкретного проекта:
.claude/skills/my-skill/SKILL.md
Для  специфичных  задач:  деплой  именно  этого  сервера  с  его  конфигурацией,
работа с этой конкретной базой данных, сборка именно этого проекта. Зашёл в
другую папку — скилла нет, и это правильно, потому что он там не нужен.
Когда  какой  выбирать? Если  скилл  привязан  к  конкретному  проекту  —
проектный.  Если  пригодится  везде  —  глобальный.  Не  уверен  —  начни  с
глобального, потом перенесёшь если нужно.
Агент автоматически сканирует обе директории при запуске.
Как скилл запускается
Способ 1: Автоматически — по словам в чате. Модель видит описания всех
скиллов.  Ты  пишешь  "подними  n8n  на  сервере"  → модель  находит  скилл,  в
description которого есть "подними", "n8n", "сервер" → активирует его. Ты даже не
знаешь, что скилл сработал — просто получаешь качественный результат.
Плюсы: удобно, не нужно помнить команды.
Минусы:  не  100%  надёжно.  Короткие  или  нестандартные  запросы  могут  не
триггернуть. Иногда модель выбирает не тот скилл.

Способ 2: Слэш-команда — явный запуск. В чате набираешь /  — появляется

11. Тестируй, ломай, чини

Три вещи, которые нужно проверить
1. Триггерится ли скилл?
Самая частая проблема — скилл создан, но не активируется. Или активируется
когда не нужно. Проверяем на примере скилла review-reply (ответы на отзывы):
Запрос Должен сработать?Результат
"Ответь на отзыв клиента" Да
"Вот отзыв: отличный кофе, приду ещё"Да
"Напиши ответ на негативный отзыв в 2ГИС"Да
"Напиши стихотворение про кофе" Нет
"Какая погода завтра" Нет
Запусти 5-6 таких запросов. Если скилл молчит на "ответь на отзыв" — description
слишком узкий. Если срабатывает на "напиши стихотворение" — слишком общий.

Скилл триггернулся — но выдал что нужно? Чеклист:
Формат правильный? КП содержит все разделы из шаблона? Конспект имеет
тайминги?

правильное?
Тон правильный? Ответ на отзыв — тёплый и конкретный, а не "Уважаемый
клиент, благодарим за обращение"?
Маршрутизация работает? Попроси "покажи прайс" — скилл открыл price-
list.md, а не шаблон КП?

Deploy Workflow — 6 фаз

14. Скрипты: когда модели нужны руки
Большинству скиллов скрипты не нужны. Если скилл пишет тексты, анализирует
данные, составляет отчёты — хватит одного SKILL.md.
Скрипты нужны когда:
Сложная bash-логика (установка софта, конфигурация)
Нужна идемпотентность (безопасно запустить повторно)
Нужна предсказуемость (одни параметры → один результат)
Обращение к внешним API (Supabase, системы перевозчиков, CRM)
В devops-ak каждый скрипт следует четырём правилам:
Идемпотентность.install-n8n.sh — если контейнер уже работает, не
трогает. Безопасно запустить 5 раз.
Генерация credentials. Пароли через openssl rand -hex 16, не
"Password123".
JSON-лог. Результат в /root/logs/ — скилл читает для отчёта.
Exit code. 0 = успех, остальное = ошибка.
Почему скрипты, а не команды от модели?
Мы  начинали  без  скриптов.  Модель  сама  генерировала  Docker  Compose.
Проблемы:
Каждый раз compose чуть отличался (модель "творчески" подходила)
Забывала healthcheck или restart policy
Пароли вида "n8n_password_2024"
Повторный запуск ломал предыдущую установку
С  готовыми  скриптами:  один  compose,  протестирован  50  раз,  пароли
криптостойкие. Модель вызывает — скрипт делает.
Правило: текстовый скилл → скрипты не нужны. Скилл работает с системой
→ напиши скрипты для критичных операций.

Словарь терминов
Agent  Skill —  набор  инструкций  в  формате  SKILL.md,  обучающий  AI-агента
конкретной задаче. По сути — структурированный рецепт для ИИ.
API (Application Programming Interface) — интерфейс, через который программы
общаются  друг  с  другом.  Например,  скрипт  отправляет  запрос  к  API  Supabase  и
получает данные из базы. Как "окошко выдачи" в фастфуде — ты говоришь что нужно,
получаешь результат.
Bash —  командная  оболочка  (терминал)  в  Linux/macOS.  Скрипты  на  Bash  —  это
текстовые  файлы  с  командами,  которые  выполняются  последовательно.  Как
автоматический набор действий в терминале.
CLAUDE.md — файл с правилами проекта, который агент читает при каждом запуске.
В отличие от SKILL.md, загружается всегда, а не по запросу.
Credentials — учётные данные: логины, пароли, API-ключи, токены доступа. Всё, что
нужно для авторизации в сервисе.
DNS  (Domain  Name  System) —  система,  которая  превращает  доменное  имя
(example.com) в IP-адрес сервера (185.x.x.x). Как телефонная книга интернета.
Docker — платформа для запуска приложений в изолированных контейнерах. Каждый
сервис (база данных, веб-сервер) работает в своём контейнере, не мешая другим.
Docker Compose — инструмент для запуска нескольких Docker-контейнеров одной
командой.  Описываешь  все  сервисы  в  одном  файле  docker-compose.yml и
поднимаешь разом.
Generic скилл — скилл, который даёт агенту широкую свободу действий (сам пишет
SQL,  сам  решает  как  подступиться).  Противоположность  конкретного  скилла,  где
действия зашиты в скрипты.
Git  /  GitHub —  система  контроля  версий  (Git)  и  платформа  для  хранения  кода
(GitHub). Позволяет отслеживать изменения, работать в команде, хранить проекты.
JSON (JavaScript Object Notation) — формат хранения данных в виде пар "ключ:
значение".  Читается  и  людьми,  и  программами.  Пример:
{"name": "Иван", "age": 30}.
Kebab-case —  стиль  именования,  где  слова  пишутся  строчными  и  разделяются
дефисами: my-skill-name. Стандарт для имён скиллов.
Markdown — простой язык разметки текста. #  — заголовок, жирный, - список.
SKILL.md написан на Markdown.

Маршрутизация — секция в SKILL.md, которая направляет разные типы запросов к
разным файлам. Как оглавление книги: "если про бэкапы — читай главу 3".
MCP  (Model  Context  Protocol) —  протокол  подключения  внешних  систем  (баз
данных,  API,  SaaS-сервисов)  к  AI-агенту.  Как  USB-разъём:  стандартный  способ
подключить что угодно.
Контекстное окно — "оперативная память" модели. Максимальный объём текста (в
токенах), который модель может обработать за раз. Включает инструкции, диалог и
результаты работы.
Крон (cron) — планировщик задач в Unix/Linux. Позволяет запускать команды по
расписанию: каждый час, каждый день в 8:00, каждый понедельник. Используется
для автоматизации скиллов.
Идемпотентность — свойство операции, которую можно выполнить повторно без
нежелательных последствий. Запустил скрипт 5 раз подряд — результат тот же, что
после первого раза.
OpenClaw — автономный AI-агент, который работает на сервере 24/7 и общается
через Telegram. Поддерживает тот же формат скиллов (SKILL.md).
Оркестрация — координация работы нескольких скиллов или агентов для решения
сложной задачи. Как дирижёр: каждый инструмент играет своё, а вместе — симфония.
Поэтапная загрузка — принцип, при котором инструкции загружаются не все сразу,
а  по  мере  необходимости.  Сначала  метаданные,  потом  SKILL.md,  потом  нужный
reference-файл.
Промпт — текстовый запрос к AI-модели. "Напиши письмо клиенту" — это промпт.
Системный промпт — начальная инструкция, определяющая поведение модели.
Scheduled  Tasks —  запланированные  задачи  в  Claude  Code.  Скилл  запускается
автоматически по расписанию, без участия пользователя.
Скрипт  (script) —  текстовый  файл  с  набором  команд,  которые  выполняются
автоматически. Скрипт  install-n8n.sh — это последовательность bash-команд для
установки n8n.
Слэш-команда —  команда,  начинающаяся  с  / .  Ввод  /devops-ak в  чате
гарантированно запускает скилл devops-ak. Способ явного запуска скилла.
SSH (Secure Shell) — протокол безопасного удалённого подключения к серверу.
Через SSH ты управляешь сервером из своего терминала, как если бы сидел за ним.
SSL —  сертификат  безопасности  для  сайта.  Обеспечивает  шифрование  (https://
вместо http://). Без SSL — браузер показывает "небезопасно".

Токен — единица текста для LLM. Примерно ~4 символа английского текста или ~1-2
символа русского. Чем больше токенов занимают инструкции — тем меньше места для
работы.
TodoWrite — встроенный инструмент Claude Code для отслеживания задач. Модель
создаёт чеклист и отмечает выполненные шаги — удобно для длинных многошаговых
процессов, где легко потерять прогресс.
Триггер —  слово  или  фраза  в  описании  скилла,  по  которой  модель  решает  его
активировать.  Пользователь  пишет  "подними  сервер"  → модель  видит  триггер
"подними" → запускает скилл.
Train/test  split —  разделение  данных  на  обучающую  и  тестовую  выборки.  В
контексте skill-creator: 60% тест-кейсов используются для оптимизации description,
40% — для проверки результата.
Файрвол  (UFW) —  программа,  которая  контролирует  сетевой  доступ  к  серверу.
Разрешает  подключения  только  по  определённым  портам  (например,  22  для  SSH,
80/443 для веб).
VPS (Virtual Private Server) — виртуальный сервер в облаке. Арендуешь у хостера
— получаешь полноценный компьютер с Linux, доступный 24/7 по SSH.
Воркфлоу (workflow) — последовательность шагов для выполнения задачи. Скилл
по сути и есть описание воркфлоу: шаг 1, шаг 2, проверка, результат.
